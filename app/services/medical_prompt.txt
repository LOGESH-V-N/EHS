## DOCUMENT TYPE CLASSIFICATION (MANDATORY FIRST STEP)

**CRITICAL**: The `document_type` field MUST be EXACTLY ONE of these 12 values (case-sensitive):

1. `Discharge_Summary`
2. `Patient_Review_And_Monitoring`
3. `Patient_Contacts_EHS`
4. `Appointment_Letter`
5. `Admission_Letter`
6. `111_First_ED_Report`
7. `Ambulance_Service`
8. `Medication_Request`
9. `Death_Notification`
10. `Treatment_Plan`
11. `Did_Not_Attend_DNA`
12. `Miscellaneous`

### Classification Decision Tree:

**Step 1**: Scan for discharge-specific keywords
- IF contains: "discharged", "discharge summary", "hospital discharge", "discharge date", "inpatient treatment summary"
- THEN → `Discharge_Summary`

**Step 2**: Check for ongoing monitoring
- IF contains: "follow-up review", "clinic visit", "monitoring table", "serial observations", "diabetes monitoring", "regular review"
- AND NOT discharge-related
- THEN → `Patient_Review_And_Monitoring`

**Step 3**: Check for emergency helpline
- IF contains: "NHS Direct", "emergency helpline", "EHS contact", "triage call", "out-of-hours service"
- THEN → `Patient_Contacts_EHS`

**Step 4**: Check for appointment notification
- IF document is primarily: date + time + location for scheduled visit
- AND minimal clinical details
- THEN → `Appointment_Letter`

**Step 5**: Check for pre-admission instructions
- IF contains: "pre-admission", "what to bring", "fasting requirements", "before hospital admission"
- THEN → `Admission_Letter`

**Step 6**: Check for 111 service
- IF contains: "NHS 111", "111 service", "case reference", "pathways assessment", "ED referral from 111"
- THEN → `111_First_ED_Report`

**Step 7**: Check for ambulance service
- IF contains: "ambulance incident", "paramedic", "pre-hospital care", "ambulance crew", "vital signs in ambulance"
- THEN → `Ambulance_Service`

**Step 8**: Check for medication focus
- IF document is primarily: prescription request, medication changes, repeat prescription
- THEN → `Medication_Request`

**Step 9**: Check for death documentation
- IF contains: "death certificate", "cause of death", "time of death", "medical certificate"
- THEN → `Death_Notification`

**Step 10**: Check for treatment plan
- IF contains: "care plan", "treatment strategy", "management approach", "treatment goals"
- THEN → `Treatment_Plan`

**Step 11**: Check for Did Not Attend notification
- IF contains: "did not attend", "DNA", "failed to attend", "non-attendance", "missed appointment", "did not attend their appointment", "was not present"
- AND document is primarily a notification about a missed or failed attendance
- THEN → `Did_Not_Attend_DNA`

**Step 12**: FALLBACK — Miscellaneous (LAST RESORT ONLY)
- IF the document does NOT clearly match ANY of Steps 1–11 above
- AND you have genuinely re-checked all 11 types and none fit
- THEN → `Miscellaneous`

### IMPORTANT RULES FOR Miscellaneous:
- `Miscellaneous` is NEVER a first choice — it is only used when ALL 11 other types have been ruled out
- Do NOT use `Miscellaneous` because the document is unclear or partially readable — attempt classification with available text
- Do NOT use `Miscellaneous` for documents that loosely match a type — if it is roughly a treatment plan, use `Treatment_Plan`
- Examples of documents that correctly use `Miscellaneous`:
  - Internal hospital administrative memos
  - GP-to-GP correspondence that is purely informational (no clinical action, no appointment, no discharge)
  - Consent forms
  - Insurance or legal medical reports
  - Safeguarding referral documents
  - Any NHS document that is genuinely not classifiable into the 11 types above

### 2A. CONSULTANT NAME (sender_information.name) — STRICT PRIORITY RULES

NHS letters always contain MULTIPLE doctor names. You MUST apply this priority order:

**PRIORITY 1 — Signature Block (HIGHEST PRIORITY)**
The correct consultant is the one who **SIGNED** the letter.
Look for:
- A name appearing AFTER the letter body, near the bottom
- Patterns: "Yours sincerely,", "Kind regards,", "Signed:", followed by a name
- A name printed below a signature line
- Format often: "Dr. [Name]" then "Consultant in [Specialty]" on the next line
→ THIS person is the `sender_information.name`

**PRIORITY 2 — Letterhead "From" Section**
If there is no clear signature, look for:
- "From: Dr. [Name]"
- A name printed in the letterhead alongside the department
→ Use this as `sender_information.name`

**PRIORITY 3 — Dictated/Authored By**
If neither above exists, look for:
- "Dictated by:", "Letter prepared by:", "Author:"
→ Use this as `sender_information.name`

**NAMES TO EXPLICITLY IGNORE (DO NOT use these as sender_information.name):**
- The GP/recipient name — usually appears after "Dear Dr. [GP Name]," or in the "To:" section
- Names in "cc:", "Copy to:", "Copies to:" lists
- Referring consultant names — usually "referred by Dr. [Name]" or "referral from Dr. [Name]"
- Names in the patient's medical history, e.g., "previously seen by Dr. [Name]"
- Any name that appears BEFORE "Dear" — that is the recipient

**CRITICAL VALIDATION — BEFORE YOU OUTPUT sender_information.name:**
Ask yourself:
1. Is this the person who WROTE and SIGNED this letter?
2. Does this person have a designation/department that matches the sending hospital section?
3. Is this person listed in "cc:" or "Dear Dr."? → If YES, you have the WRONG person. Go back and re-read.

#### STEP-BY-STEP EXTRACTION PROCESS (follow in order):

**STEP 1 — Identify who signed the letter first (sender_information.name)**
You must know the signer before you can find their department. Complete Section 2A first.

**STEP 2 — Look directly below the signer's name in the sign-off block**
The sign-off block is at the BOTTOM of the letter, after "Yours sincerely," or "Kind regards,".
The department is almost always printed on the line(s) immediately below the consultant's name and title.

Example sign-off block:
```

Consultant Cardiologist

```
→ department = "Department of Cardiology"

**STEP 3 — If the sign-off block has no department line, look in the LETTERHEAD**
The letterhead is the printed header at the very TOP of the document (hospital name, address, logo area).
A department name is often printed in the letterhead alongside the hospital name.

Example letterhead:
```

RESPIRATORY MEDICINE              ← THIS is the correct department if sign-off has none
London Road, Reading, RG1 5AN
```
→ department = "Respiratory Medicine"

**STEP 4 — If still not found, look for a "From:" or "Re:" line near the top**
Some NHS letters have a structured header section:
```
From: Department of Orthopaedics, St. Mary's Hospital

```
→ department = "Department of Orthopaedics"

**STEP 5 — If none of the above contain a department, set department = ""**
Do NOT guess. Do NOT use a department found anywhere else in the document if it cannot be confirmed as belonging to the signer.

##  NHS NUMBER — STRICT VALIDATION RULES (CRITICAL)

**What is an NHS Number?**
An NHS Number is a **unique 10-digit identifier** assigned to every NHS patient in England, Wales, and Isle of Man.

###  NHS Number Format Rules (ALL must be true):
1. **Exactly 10 digits** — no more, no less
2. **Displayed as**: `XXX XXX XXXX` (3 digits, space, 3 digits, space, 4 digits) — but may appear without spaces
3. **Explicitly labeled** in the document as: `"NHS Number"`, `"NHS No"`, `"National Code"`,  `"NHS No."`, `"NHS #"`, or `"NHS Num"`
4. **Never alphanumeric** — NHS numbers contain digits ONLY, no letters

###  Numbers that are NOT NHS Numbers — NEVER extract these as `nhs_number`:
- **Hospital Number / MRN**: Shorter, often 6–8 digits, may be alphanumeric (e.g., `A123456`, `MRN: 789012`). Hospital numbers are frequently labeled "Hospital No", "Hosp No", "Patient No", "MRN", "URN", or similar — NEVER use these as nhs_number.
- **Phone / Mobile**: 11 digits starting with 07, 01, or 02 (e.g., `07700 900123`)
- **Referral Number**: Often alphanumeric (e.g., `REF/2024/001`)
- **Case Reference**: Mixed letters + digits (e.g., `CR-2024-00145`)
- **Ward / Bed Number**: Very short number

###  Strict Extraction Rules:

```
IF a number is explicitly labeled "NHS Number" / "NHS No" / "NHS No. / "National Code" AND is exactly 10 digits:
   → Extract it as nhs_number

IF a number is labeled "Hospital No", "Hosp No", "MRN", "Patient No", "URN", or similar:
   → This is a HOSPITAL NUMBER — set nhs_number = "" (DO NOT use this)

IF a number is NOT explicitly labeled as "NHS Number":
   → Set nhs_number = "" (DO NOT guess or infer)

IF nhs_number appears redacted (e.g., "XXX XXX XXXX", "*** *** ****", "[REDACTED]"):
   → Set nhs_number = "" (DO NOT substitute with any other number)

IF document is missing an NHS Number entirely:
   → Set nhs_number = "" (DO NOT use phone number, MRN, hospital number, or any other number as a substitute)
```

###  Common Errors to AVOID:
```json
//  WRONG — Hospital number incorrectly used as NHS number
"nhs_number": "A123456"

//  WRONG — Phone number incorrectly used as NHS number
"nhs_number": "07700 900123"

//  WRONG — Unlabeled number assumed to be NHS number
"nhs_number": "1234567890"

//  CORRECT — NHS number is missing or redacted, field left empty
"nhs_number": ""

//  CORRECT — Valid NHS number present, explicitly labeled in document
"nhs_number": "485 777 3456"
```

## PROBLEMS SECTION — COMPREHENSIVE EXTRACTION RULES (CRITICAL UPDATES)

### What MUST be included in `problems`:

The `problems` array must capture ALL of the following categories found in the document:

#### 1. Chronic / Ongoing Medical Conditions
- Past medical history (PMH) — include ALL conditions listed under "Past Medical History", "PMH", "Background", "Medical History", or equivalent headings
- Long-standing conditions even if described as "historical" or "background"
- **IMPORTANT**: Past medical history IS relevant and MUST be included in problems. Do not omit PMH entries.

#### 2. Active Presenting Symptoms (NEW — REQUIRED)
- Symptoms the patient is experiencing NOW that prompted this clinical encounter
- Examples: "Vomiting", "Nausea", "Severe abdominal pain", "Chest pain", "Shortness of breath", "Elevated D-Dimer", "Severe gallbladder flare up"
- These MUST be listed in problems even if they are not formally diagnosed conditions
- Look for symptoms described in the presenting complaint, history of presenting complaint, or reason for referral sections

#### 3. Allergies (NEW — REQUIRED)
- ALL documented allergies must be listed in the `problems` array
- Label them clearly, e.g., "Allergy: Penicillin", "Allergy: Latex"
- Look for sections labeled "Allergies", "Drug Allergies", "Known Allergies", "Adverse Reactions"
- If "No Known Allergies" or "NKDA" is stated, do NOT add an entry

#### 4. Surgical History (NEW — REQUIRED)
- Previous surgical operations must be included in problems
- Examples: "Previous appendectomy", "Previous cholecystectomy", "Hip replacement surgery"
- Look for sections labeled "Surgical History", "Previous Operations", "Past Surgical History"

#### 5. Claustrophobia and Other Procedural Contraindications
- Conditions that affect clinical management or procedure selection must be in problems
- Example: "Claustrophobia" (relevant for MRI), "Latex allergy" (relevant for surgery)

### What MUST NOT be in `problems`:
- Mobility aids or adaptive equipment (e.g., "Using a walking stick to support mobility" — this is a functional status note, NOT a medical problem)
- Generic lifestyle descriptions that are not medical diagnoses
- Normal physiological findings

### PROBLEMS vs DIAGNOSIS — CLARIFICATION:
- `problems`: Pre-existing conditions, background history, PMH, symptoms, allergies, surgical history
- `diagnosis`: New diagnoses made DURING THIS CLINICAL EPISODE that are formally confirmed in this document

### CRITICAL — PRESENTING COMPLAINT / REFERRAL REASON vs DIAGNOSIS:
A presenting symptom (e.g., "abdominal pain", "chest pain") that prompted the referral may appear both as the referral reason AND under a "Diagnosis:" heading in the document. Apply this rule strictly:

- **If the document has a "Diagnosis:" section and lists the condition there** → place it in `diagnosis` ONLY. Do NOT also add it to `problems` as a symptom or presenting complaint.
- **If the condition is ONLY mentioned as a referral reason or presenting complaint, with NO formal diagnosis heading confirming it** → place it in `problems` ONLY as an active symptom.
- **The referral reason alone does NOT qualify something for `problems`** when it has been formally diagnosed and listed under "Diagnosis:" in the document.
- **ONE section only — never both.** If in doubt, `diagnosis` takes priority over `problems` when the document explicitly labels it as a diagnosis.

### CRITICAL — VERBATIM EXTRACTION FROM "Diagnosis:" HEADING (NO PARAPHRASING):
When a document contains an explicit **"Diagnosis:"** section or heading:
- Extract each listed item **verbatim** (word-for-word) into `diagnosis`
- Do NOT shorten, rephrase, or paraphrase the diagnosis text when copying it to `diagnosis`
- Do NOT then independently re-extract the same condition into `problems` using different or shortened wording — this causes duplicates with mismatched text which breaks SNOMED matching
- **The wording in `diagnosis` must exactly match the wording in the document's "Diagnosis:" section**

### CRITICAL — PMH ITEMS LISTED WITHIN A "Diagnosis:" SECTION:
Sometimes a document's "Diagnosis:" list contains a mix of new diagnoses AND past medical history items (e.g., "Previous history of CA bowel", "Known hypertension"). Apply this rule:
- If an item in the "Diagnosis:" list contains words like: "previous", "previous history of", "known", "background", "history of" → it is a **past medical history item** → place it in `problems` ONLY, NOT in `diagnosis`
- All other items in the "Diagnosis:" list without such PMH indicators → place in `diagnosis` ONLY
- **NEVER place a PMH item in both `problems` and `diagnosis`**

### Example — mixed "Diagnosis:" list with PMH item:
```
Document "Diagnosis:" section:
1. Degenerative change right hip with right antalgic gait secondary to this
2. Some intermittent left leg radicular symptoms with some narrowing around the left L5 nerve root in the lateral recess at L4/5
3. Previous history of CA bowel but no known metastatic disease

→ diagnosis: items 1 and 2 (verbatim, no rephrasing)
→ problems: item 3 only (PMH — contains "Previous history of")
→ problems does NOT also contain items 1 or 2 in any form
```

## MEDICATIONS EXTRACTION (ENHANCED RULES)

### Classification Rules:

## ACTIONS EXTRACTION (CRITICAL DEFINITIONS)

### Follow-up vs. Appointment vs. Action

**FOLLOW-UP** (Clinical intention):
- Definition: Future review or reassessment recommendation, or referral to a service
- Does NOT require confirmed booking
- Examples:
  - "Review in cardiology clinic in 6 weeks"
  - "Follow-up with GP in 2 weeks"
  - "Recommend referral to Physiotherapy"
  - "Refer to asthma service"
  - "Arrange oncology review"

**APPOINTMENT** (Confirmed booking with date/time):
- Definition: Scheduled visit with explicit confirmation AND a date and/or time
- MUST have a confirmed date/time to be classified as appointment
- Examples:
  - "Appointment scheduled for 15th March 2024 at 2pm"
  - "Booked to see Dr. Smith on Tuesday 14th January 2025 at 10:00"
- Counter-examples (NOT appointments — classify as follow-up):
  - "Please arrange appointment" (action, not appointment)
  - "To be seen in clinic" (follow-up, not appointment)
  - "Appointment booked" with NO date/time given → classify as follow-up

**ACTION** (GP/Clinician-directed instruction ONLY):
- Definition: A specific task or instruction directed **at the GP or another clinician**. Instructions directed at the patient MUST NEVER appear here.
- INCLUDE — instructions where the GP, clinician, or clinical team is the actor:
  - "GP to arrange blood tests"
  - "GP to review medication in 4 weeks"
  - "Prescribe [medication] — request to GP"
  - "Arrange MRI scan within 2 weeks"
  - "Book physiotherapy appointment"
  - "Please arrange an urgent referral"
  - "GP to check blood pressure at next visit"
  - "Inquiries regarding medication to the doctor" (GP medication inquiry actions MUST be captured here)
  - Any request or task where a clinician or clinical system is performing the action
- EXCLUDE — NEVER place these in actions.actions:
  - Patient-directed instructions: "Patient to collect prescription from pharmacy", "Patient to monitor symptoms at home"
  - Patient self-care advice: "Take medication with food", "Avoid strenuous activity for 6 weeks", "Rest at home"
  - Lifestyle or health education advice addressed to the patient
  - Sentences using patient-addressed language: "You should…", "You need to…", "Please ensure you…", "Make sure you…"
  - Any instruction where the subject performing the action is the **patient** — it does NOT belong in actions.actions regardless of clinical relevance
  - If in doubt whether an instruction is GP-directed or patient-directed: if the patient is the actor → EXCLUDE

### Extraction Rules:
1. **DO NOT convert actions into appointments**
2. **DO NOT convert advice into follow-ups**
3. **Appointments MUST have a date/time — if no date/time is present, classify as follow-up**
4. **Include the date/time in appointment_date_time field whenever it is stated**
5. **GP medication inquiries or medication-related instructions MUST be captured in actions**
6. **Referrals to services (e.g., physiotherapy, asthma service, dietitian) MUST appear in follow_up**

### FOLLOW-UP TEXT — VERBATIM EXTRACTION RULE (CRITICAL)
 
This is one of the most commonly broken rules. The `follow_up_text` field MUST be copied **word for word** from the document. You are NOT allowed to paraphrase, interpret, summarise, or add context that is not explicitly written.
 
#### WHAT THIS MEANS IN PRACTICE:
 
**PRONOUNS — DO NOT SUBSTITUTE:**
- If the document says **"you"**, write **"you"** — do NOT replace it with "patient", "patient's parents", "family", "carer", or any other noun
- If the document says **"her"**, write **"her"** — do NOT replace it with the patient's name
- If the document says **"we"**, write **"we"** — do NOT replace it with the consultant's name or department
- The reader of the JSON will have the full letter for context — you do not need to resolve pronouns
 
**DO NOT ADD WORDS THAT ARE NOT IN THE DOCUMENT:**
- If the document does not say "parents" → do NOT write "parents" in the follow_up_text
- If the document does not say "GP" → do NOT write "GP" in the follow_up_text
- If the document does not say "MRI results" but says "the results" → write "the results", not "MRI results"
- Adding inferred context (even if correct) is a violation of the verbatim rule
 
**DO NOT SUMMARISE:**
- If the document says "I will be discussing the results with you shortly" → the follow_up_text is exactly that sentence
- Do NOT shorten it to "Discuss results" or expand it to "Discuss MRI results with patient's parents"
 
#### CORRECT vs WRONG EXAMPLES:
 
**Document text:** *"I will be discussing the results with you shortly."*
```
WRONG: follow_up_text = "Discuss MRI results with patient's parents"
       Reason: "MRI results" was inferred (doc says "the results"), "patient's parents" was inferred (doc says "you")
 
CORRECT: follow_up_text = "I will be discussing the results with you shortly"
         follow_up_timeframe_or_date = "shortly"
```
 
**Document text:** *"Review in clinic in 6 weeks."*
```
WRONG: follow_up_text = "Patient to be reviewed by consultant in outpatient clinic in 6 weeks"
       Reason: "Patient", "consultant", "outpatient" are all inferred — not in the document
 
CORRECT: follow_up_text = "Review in clinic in 6 weeks"
         follow_up_timeframe_or_date = "6 weeks"
```
 
**Document text:** *"We will arrange a follow-up appointment for her."*
```
WRONG: follow_up_text = "Arrange follow-up appointment for Lucy"
       Reason: "Lucy" was inferred — the document says "her"
 
CORRECT: follow_up_text = "We will arrange a follow-up appointment for her"
```
 
#### TIMEFRAME FIELD:
- `follow_up_timeframe_or_date` should contain ONLY the timeframe or date extracted from the follow-up sentence
- Examples: "6 weeks", "shortly", "2 weeks", "3 months", "when results available"
- If no timeframe is stated, leave as `""`
